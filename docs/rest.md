# REST proxy (`protoc-gen-protobridge`)

This plugin generates a REST + WebSocket + SSE gateway in front of your gRPC backend, plus an OpenAPI 3.1 spec, an AsyncAPI 3.0 spec for streams, a `Dockerfile`, a Kubernetes manifest, and `.env` files.

```
HTTP / WebSocket / SSE clients
        │
        ▼
   ┌──────────────┐
   │  REST proxy  │  ← generated by protoc-gen-protobridge
   └──────┬───────┘
          │  gRPC + metadata
          ▼
   ┌──────────────┐
   │ gRPC backend │
   └──────────────┘
```

For the full annotation reference (HTTP routing, auth, validation, headers, query params, oneof, enums, streaming) see the main [README](../README.md). This page focuses on the plugin invocation and generated layout.

## Running the generator

```bash
protoc \
  --protobridge_out=./gateway \
  --protobridge_opt=handler_pkg=github.com/you/myapp/gateway/handler \
  -I . -I path/to/protobridge/proto -I path/to/googleapis \
  your/service.proto
```

`handler_pkg` is the **Go import path** that the emitted `main.go` uses to import the per-service handler files. If omitted, the plugin walks up from the current directory to find `go.mod` and looks for a conventional output dir (`gen/protobridge`, `protobridge`, `gen`); reproducible CI builds should pass it explicitly.

## Generated layout

```
gateway/
├── main.go              # entry point, package main
├── handler/             # one file per gRPC service, package handler
│   ├── chat_service.go
│   └── auth_service.go
├── Dockerfile
├── k8s.yaml
├── .env.example
├── .env.defaults
└── schema/
    ├── openapi.yaml
    └── asyncapi.yaml    # only if you have streaming methods
```

`main.go` and the deployment manifests sit at the root; Go sources are isolated under `handler/` so the directory stays browsable.

## Build + run

```bash
cd gateway
go build -o gateway .
PROTOBRIDGE_CHAT_SERVICE_ADDR=localhost:50051 ./gateway
```

The proxy listens on `:8080` (override with `PROTOBRIDGE_PORT`). Every backend service has its own `PROTOBRIDGE_<SCREAMING_SNAKE>_ADDR` env var that resolves to a gRPC address; the connection pool scales from 1 to N connections per backend based on load.

## Environment variables

| Variable | Default | Purpose |
|---|---|---|
| `PROTOBRIDGE_PORT` | `8080` | HTTP listen port |
| `PROTOBRIDGE_<SVC>_ADDR` | (required) | gRPC backend address per service |
| `PROTOBRIDGE_<SVC>_TLS` | `false` | Use TLS for the gRPC connection |
| `PROTOBRIDGE_<SVC>_GRPC_OPTIONS` | — | Additional gRPC dial options (key=value, comma-separated) |
| `PROTOBRIDGE_OTEL_ENDPOINT` | — | OTLP endpoint for traces + metrics |
| `PROTOBRIDGE_OTEL_SERVICE_NAME` | `protobridge` | OpenTelemetry service.name |
| `PROTOBRIDGE_SENTRY_DSN` | — | Sentry DSN for panic / error reporting |
| `PROTOBRIDGE_METRICS_PORT` | — | Standalone Prometheus metrics endpoint port (optional) |
| `PROTOBRIDGE_READ_HEADER_TIMEOUT` | `10s` | HTTP server read-header timeout |
| `PROTOBRIDGE_READ_TIMEOUT` | `30s` | HTTP server read timeout |
| `PROTOBRIDGE_WRITE_TIMEOUT` | `0` (unlimited, for WS/SSE) | HTTP server write timeout |
| `PROTOBRIDGE_TLS_CERT` / `PROTOBRIDGE_TLS_KEY` | — | Serve HTTPS instead of HTTP |

Generated `.env.example` enumerates every variable for your concrete service. `.env.defaults` ships sensible defaults.

## Health endpoint

`GET /healthz` always returns `{"status":"ok"}` — independent of backend health. Use it for Kubernetes liveness/readiness probes.
