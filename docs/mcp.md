# MCP proxy (`protoc-gen-mcp`)

This plugin generates a [Model Context Protocol](https://modelcontextprotocol.io/) server that exposes selected gRPC methods as MCP tools. The proxy is a thin translation layer — your business logic stays in the gRPC backend.

```
LLM client (Claude Desktop, Cursor, custom)
        │  stdio JSON-RPC  /  streamable HTTP
        ▼
   ┌──────────────┐
   │  MCP proxy   │  ← generated by protoc-gen-mcp
   └──────┬───────┘
          │  gRPC + metadata (session_id, auth_token, …)
          ▼
   ┌──────────────┐
   │ gRPC backend │
   └──────────────┘
```

## Annotations

```protobuf
import "protobridge/options.proto";

service ToolService {
  // Service-level opt-in: every method becomes an MCP tool unless it
  // explicitly sets (protobridge.mcp) = false.
  option (protobridge.mcp_default) = true;

  // Tool description comes from this proto comment. Returns the new task ID.
  rpc CreateTask(CreateTaskRequest) returns (CreateTaskResponse) {
    option (protobridge.mcp_scope) = "chat session";
  }

  // Per-method opt-in (when service has no mcp_default).
  rpc GitCreatePR(GitCreatePRRequest) returns (GitCreatePRResponse) {
    option (protobridge.mcp) = true;
    option (protobridge.mcp_scope) = "worker session";
  }

  // Override the description if the proto comment is unsuitable for an LLM.
  rpc Internal(Empty) returns (Empty) {
    option (protobridge.mcp) = true;
    option (protobridge.mcp_description) = "Re-syncs project metadata. Slow; use sparingly.";
  }

  // Methods without (protobridge.mcp) = true are not emitted as tools.
  rpc Hidden(Empty) returns (Empty);
}
```

| Annotation | Scope | Purpose |
|---|---|---|
| `(protobridge.mcp) = true` | method | Expose this RPC as an MCP tool |
| `(protobridge.mcp_default) = true` | service | All methods opt in; per-method `mcp = false` opts out |
| `(protobridge.mcp_scope) = "..."` | method | Free-form availability hint, appended to the tool description as `Available in: <scope>`. Documentation only — backend still enforces real scope rules |
| `(protobridge.mcp_description) = "..."` | method | Override for the LLM-facing description; falls back to the proto leading comment when empty |

Streaming methods are silently skipped — MCP tools are unary by spec.

## Tool name + description

- **Name**: `snake_case` of the gRPC method name (`CreateTask` → `create_task`).
- **Description**: `mcp_description` if set, else the proto leading comment extracted from `SourceCodeInfo`. The `mcp_scope` value (when set) is appended on its own line.
- **Input schema**: derived from the proto request message — proto scalars map to standard JSON Schema types, enums become `string` with `enum` listing `x_var_name` aliases when present, repeated fields become arrays, `google.protobuf.Empty` becomes an empty schema, oneof fields are skipped.

## Running the generator

```bash
protoc \
  --mcp_out=./gen/mcp \
  --mcp_opt=handler_pkg=github.com/you/myapp/gen/mcp/handler \
  -I . -I path/to/protobridge/proto \
  your/service.proto
```

`handler_pkg` is the **Go import path** that the emitted `main.go` uses to import the per-service handler files. As with the REST plugin, you can omit it and the plugin will walk up from the current directory to find a `go.mod` plus a conventional output dir (`gen/protobridge`, `protobridge`, `gen`) — explicit is better for CI.

Optional: `--mcp_opt=forward=session_id;auth_token` overrides the identity keys forwarded into gRPC metadata. Default is `session_id`.

## Generated layout

```
gen/mcp/
├── main.go                    # entry point, package main
├── handler/                   # one file per gRPC service, package handler
│   └── tool_service.go
└── schema/
    ├── openrpc.json           # OpenRPC 1.3.2 — for client codegen
    └── mcp-tools.json         # cached `tools/list` response — for MCP introspection
```

Connect to the backend the same way as the REST proxy (via `PROTOBRIDGE_<SERVICE>_ADDR` env vars). Then choose a transport.

## Schema artifacts for client codegen

Two schema files are emitted alongside the proxy so you don't have to boot the server to discover its tools.

### `schema/openrpc.json` — for typed client codegen

[OpenRPC](https://open-rpc.org/) is the JSON-RPC equivalent of OpenAPI. Each MCP tool is exposed as an OpenRPC method with named params (one per top-level proto field) and a typed result schema. Run [`openrpc-generator`](https://github.com/open-rpc/generator) against the file to scaffold typed wrappers in TypeScript, Go, Python, etc.

```bash
npx @open-rpc/generator generate \
  --schema gen/mcp/schema/openrpc.json \
  --type client \
  --language typescript \
  --output ./client
```

The generated client typically wraps each method in a strongly-typed function (`client.createTask({title, priority, ...})`) instead of forcing callers to hand-roll JSON-RPC envelopes. MCP's wire protocol is technically `tools/call` with `{name, arguments}` — most generated clients hide that wrapping at runtime.

### `schema/mcp-tools.json` — native MCP introspection

The same JSON shape an MCP server returns from a `tools/list` request. Wrapped as `{"tools": [...]}` so it can be served verbatim. Useful for:

- Loading into [`mcp-cli`](https://github.com/wong2/mcp-cli) / `mcp-inspector` for manual exploration without booting the proxy
- Feeding to MCP client libraries that accept tool definitions and produce typed call wrappers (`mark3labs/mcp-go`, `py-mcp`, `ts-mcp`, Anthropic SDKs)
- Diff-checking tool surface changes in PR review

Both files are deterministically ordered (alphabetical by tool name) so regeneration produces zero-diff output until the proto actually changes.

## Transports

| Transport | When to use | How to enable |
|---|---|---|
| **stdio** (default) | LLM clients spawn the binary as a subprocess (Claude Desktop, Cursor, `mcp-cli`, your custom MCP host) | Default; or `PROTOBRIDGE_MCP_TRANSPORT=stdio` |
| **streamable HTTP** | Remote MCP clients, sidecar in a container, multi-tenant deployment | `PROTOBRIDGE_MCP_TRANSPORT=http`, `PROTOBRIDGE_MCP_HTTP_ADDR=:8081` |

Both transports share the same handlers, auth pipeline, and gRPC backend connection.

## Auth + identity

MCP doesn't have a one-size-fits-all auth model the way HTTP does. The proxy's `MCPAuthFunc` decides what gRPC metadata to attach for each tool call. The default implementation forwards configurable identity keys from (in order):

1. **HTTP headers** — for streamable HTTP transport. `ServeStreamableHTTP` stashes the inbound `http.Header` on the request context so `DefaultAuthFunc` can read it.
2. **MCP `initialize` params** — sent at handshake time by the client. *Reserved* — the runtime does not currently populate `ConnectionInfo.InitializeParams`; supply your own `MCPAuthFunc` if you need to read them.
3. **`SCREAMING_SNAKE` env vars** — for stdio transport, the parent process sets them when spawning the proxy.

For example, with `--mcp_opt=forward=session_id` (the default) and `SESSION_ID=abc` set in the proxy's environment, every gRPC call from the proxy carries `session_id=abc` in metadata — and the gRPC backend reads it the same way it would from REST or any other client.

If your project needs richer auth (DB lookup, token validation, container reference resolution), drop in your own `MCPAuthFunc`:

```go
auth := func(ctx context.Context, conn mcp.ConnectionInfo) (metadata.MD, error) {
    token := conn.HTTPHeaders.Get("Authorization")
    user, err := validateToken(ctx, token)
    if err != nil {
        return nil, err
    }
    return metadata.Pairs("user_id", user.ID, "session_id", user.SessionID), nil
}
srv := mcp.NewServer("my-server", "1.0.0", auth)
```

## Architecture rationale

MCP doesn't replace gRPC — it *adds* a wire format on top. Putting the MCP server inside your gRPC backend would force you to import a JSON-RPC library, manage stdio framing, and reimplement scope-based tool visibility. Putting it in a generated proxy keeps the backend pure gRPC and lets you scale MCP separately (e.g. one MCP container per session container, while the gRPC backend stays a singleton).

Scope enforcement is **always** the backend's job — `mcp_scope` is a hint for the LLM client, not an access-control mechanism. The proxy forwards every authenticated tool call; the backend decides whether the session is allowed to do what the tool claims.
